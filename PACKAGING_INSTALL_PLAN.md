# 可安装版交付计划

## 当前状态

当前仓库已经达到“开发态可用”：

- 可以启动本地服务
- 可以提交 B 站链接任务
- 可以下载音频、转写、摘要、导出结果
- 可以把任务、结果、事件写入 SQLite

但它还不是“可安装版产品”。目前仍依赖：

- 本机已有 Python
- 本机已有 ffmpeg
- 手动 `pip install`
- 手动运行脚本
- 手动配置 `.env`

这和最终目标之间还有明显差距。

## 最终交付目标

最终应交付为一套可安装版本地产品：

- Windows：`MSI`
- macOS：`DMG`
- Linux：`AppImage`

安装后的用户体验目标：

- 双击安装
- 首次启动即可运行本地服务
- 不要求用户手动安装 Python
- 不要求用户手动安装 pip 依赖
- 尽量不要求用户单独配置 ffmpeg
- 本地 UI 可查看任务和结果
- 浏览器扩展只作为辅助入口

## 当前到安装版的关键缺口

### 1. 后端还没有独立可分发产物

当前服务需要：

- `python -m video_sum_service`

后续需要：

- 可直接运行的后端二进制
- 内置依赖
- 配套资源目录

### 2. 还没有本地桌面 UI

安装版不能只交一个后台服务。

需要至少有：

- 桌面主窗口
- 服务状态页
- 任务提交页
- 历史记录页
- 结果查看页
- 设置页

### 3. 模型与资源交付未处理

当前真实链路依赖：

- `faster-whisper`
- Hugging Face 模型下载
- `ffmpeg`
- `yt-dlp`

安装版需要解决：

- ffmpeg 随包还是首启下载
- Whisper 模型按需下载还是内置最小模型
- 缓存目录和模型目录的统一管理

### 4. 配置交付方式仍偏开发态

当前依赖 `.env`。

安装版应改成：

- 安装后自动生成配置
- 桌面 UI 可修改配置
- API Key 尽量保存到本地用户目录

### 5. 服务生命周期未产品化

当前服务是命令行启动。

安装版需要：

- 桌面应用启动时拉起服务
- 应用退出时可选择保持后台运行
- 支持最小化到托盘

## 推荐安装版架构

### 桌面壳

- `Tauri + React`

职责：

- 承载本地 UI
- 管理后端服务进程
- 管理配置
- 提供安装包产物

### 后端服务

- Python FastAPI
- 使用 PyInstaller 打包为独立可执行文件

职责：

- 接收任务
- 执行处理链路
- 写入 SQLite
- 输出结果和事件

### 扩展

- 保持薄层
- 调本地服务 HTTP API

## 安装版优先路线

### 阶段 A：把后端服务做成独立可执行文件

目标：

- 不需要 Python 环境
- 双击或由桌面壳启动即可运行

任务：

- 增加 PyInstaller spec
- 固定资源目录
- 处理 ffmpeg 路径发现
- 处理模型缓存目录

### 阶段 B：补本地 Web UI 或桌面 UI

目标：

- 不再依赖脚本提交任务

任务：

- 先做本地 Web UI
- 再接入 Tauri

### 阶段 C：做桌面安装包

目标：

- 产出 MSI / DMG / AppImage

任务：

- Tauri 配置
- 安装器资源
- 图标、版本号、应用名

### 阶段 D：处理首次启动体验

目标：

- 用户安装后第一次打开就能理解怎么用

任务：

- 首次启动向导
- 模型下载提示
- API Key 配置页
- ffmpeg 检查与修复

## 当前建议的开发原则

从现在开始，所有新增能力都尽量满足这三个条件：

1. 不依赖开发机目录结构
2. 不假设用户本机已经装 Python
3. 不把关键配置只放在 `.env`

## 下一步最应该做的事

1. 固化服务的数据目录、模型目录、日志目录
2. 设计服务可执行文件的启动参数
3. 创建 `packaging/pyinstaller` 骨架
4. 开始本地 UI

## 一句话判断

当前版本已经是“开发态可用”，下一阶段的核心目标不再是单纯加功能，而是把它收敛成“可安装、可启动、可维护的本地应用”。
