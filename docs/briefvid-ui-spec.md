# BriefVid UI 规范书

## 1. 文档定位

- 适用范围：`apps/desktop` 当前 Electron 桌面端界面，以及后续同步到 Web 的同源交互页面。
- 规范目标：统一视觉语言、组件行为、文案风格和实现约束，减少“页面各做各的”的漂移。
- 实现基线：当前代码实现以 `apps/desktop/src/app.css`、`apps/desktop/src/App.tsx`、`apps/desktop/src/components/TitleBar.tsx`、`apps/desktop/src/components/UpdateDialog.tsx` 为准。
- 规范优先级：
  1. 本文档
  2. 代码中的设计 token 与组件实现
  3. 概念提案文档 `docs/briefvid-light-ui-spec.md`

## 2. 产品体验原则

### 2.1 主流程优先

- BriefVid 的核心路径是“输入链接 -> 本地处理 -> 查看结果”。
- 主操作区必须始终比概览数据、系统信息、辅助信息更显眼。
- 首屏不能让用户先理解系统架构，应该先能找到“开始处理”的入口。

### 2.2 轻专业，而不是重后台

- 风格应保持轻量、干净、可信，避免传统运维后台那种厚重、密集、冷硬的观感。
- 视觉上允许有消费级产品的精致感，但不能用大面积花哨效果抢走功能注意力。

### 2.3 状态透明

- 所有异步流程必须让用户知道“正在发生什么、是否成功、下一步能做什么”。
- 成功、运行中、失败、等待中必须有稳定且一致的状态表达。

### 2.4 双主题一致

- 浅色主题是默认工作模式，深色主题是等价主题，不是临时覆盖层。
- 新组件必须同时定义浅色和深色下的背景、文字、边框与状态色表现。

### 2.5 可扩展

- 规范不仅服务首页，也服务设置页、详情页、弹窗、任务进度、日志与未来新模块。
- 新页面应优先复用现有壳层、卡片、表单、状态与层级规则，而不是重新发明一套 UI。

## 3. 视觉系统

### 3.1 品牌气质

- 品牌主色来源于 BriefVid 图标，核心色为 `#FB7299`。
- 整体基调是暖粉珊瑚系，配合低饱和浅底和克制阴影。
- 品牌色用于主 CTA、关键状态强调、选中态和轻量点缀，不应用于大面积铺底。
- 平台品牌色只用于识别性标签，不可反向主导页面主题；BriefVid 自身品牌色仍然是全局第一视觉锚点。

### 3.3 字体与字级

- 无衬线字体：`Inter`, `Plus Jakarta Sans`, `Manrope`, `PingFang SC`, `Noto Sans SC`, `Microsoft YaHei`, `sans-serif`
- 等宽字体：`JetBrains Mono`, `SF Mono`, `Consolas`, `monospace`

| 标签与状态 | `11px` 到 `12px` | `600` 到 `700` |
| 关键数字 | `24px` 到 `32px` | `700` |

### 3.4 圆角、阴影、间距

#### 圆角

| Token | 值 | 用途 |
| --- | --- | --- |
| `--radius-sm` | `8px` | 小按钮、小标签、小关闭按钮 |
| `--radius-md` | `12px` | 普通按钮、输入框、导航项 |
| `--radius-lg` | `16px` | 中型卡片、面板 |
| `--radius-xl` | `20px` | 常规内容卡 |
| `--radius-2xl` | `24px` | 页面头、弹窗 |
| `--radius-3xl` | `32px` | 大块品牌或展示区域 |

#### 阴影

- 浅色主题使用轻阴影，避免悬浮感过重。
- 深色主题阴影可以更深，但不能让组件像漂浮在黑幕上。
- 推荐优先使用 `--shadow-sm` 到 `--shadow-lg`，不要随意定义新的超大阴影。

#### 间距

- 使用 4px 基础网格。
- 推荐节奏：`4 / 8 / 12 / 16 / 20 / 24 / 32 / 40`。
- 页面块级间距默认使用 `24px`，大区块可用 `28px` 到 `32px`。

### 3.5 动效

- 快速反馈：`120ms`
- 常规切换：`180ms`
- 稍长过渡：`240ms`
- 页面或弹窗进入动画可使用 200 到 300ms，避免超过 350ms。
- 动效原则：服务理解，不服务炫技。

## 4. 布局规范

### 4.1 应用壳层

| 区域 | 规则 |
| --- | --- |
| 顶部标题栏 | 高度 `40px`，固定在顶部 |
| 左侧边栏 | 宽度 `248px`，固定定位 |
| 主内容区 | 自边栏右侧展开，垂直滚动 |
| 常规内容容器 | `max-width: 1400px`，默认内边距 `24px 32px 40px` |
| 设置页容器 | `max-width: 1560px`，横向留白略大 |

### 4.2 页面结构模板

#### 首页 / 视频库页

- 由 `page header + hero / 概览 + 列表工具栏 + 视频网格` 组成。
- 主操作和概览应在首屏内可见。
- 搜索与筛选必须在列表标题附近，不要脱离内容区单独漂浮。

#### 设置页

- 采用 `左侧设置导航 + 右侧内容区` 的双栏布局。
- 左栏承载分组导航、品牌说明、保存入口和摘要信息。
- 右栏承载类别区块、表单、说明和环境信息。

#### 详情页

- 顶部为内容摘要区或封面区。
- 中部为主信息流。
- 右侧可使用辅助信息栏承载状态、元信息、操作或结果摘要。

#### 弹窗 / 覆盖层

- 遮罩使用半透明深色与轻模糊。
- 对话框宽度控制在 `540px` 左右，保留移动端下的缩窄能力。

### 4.3 响应式断点

| 断点 | 规则 |
| --- | --- |
| `1260px` | 设置页收紧为 `280px + 主内容` |
| `1200px` | 首页双栏区改为单栏，内容容器缩窄 |
| `1024px` | 设置页变为单栏 |
| `900px` | 侧栏从固定侧栏切换为页面顶部块 |
| `768px` | 视频网格最小卡宽降到 `200px`，局部表单改单列 |
| `640px` | 弹窗底部按钮纵向堆叠 |

## 5. 组件规范

### 5.1 标题栏

- 使用自定义窗口栏，左侧显示品牌，右侧为主题、最小化、最大化、关闭。
- 关闭按钮 hover 可使用系统红色，其他按钮保持中性。
- 图标按钮必须提供 `aria-label`。
- 拖拽区与非拖拽区必须明确分离。

### 5.2 侧边栏与导航

- 分组标签使用大写小字距，作为信息结构而不是视觉主角。
- 导航项采用图标容器 + 文案双列结构。
- 激活态使用弱品牌底和品牌文字色，不使用强烈纯色填充。
- 底部状态面板承载服务状态、版本、运行设备等低频但重要信息。

### 5.3 页面头

- 页面头用于定义“当前页面是什么、现在能做什么”。
- 结构为 `eyebrow / h2 / 一段说明文案`。
- 页面头应有轻边框与轻阴影，保持信息聚焦。

### 5.4 表单输入

- 所有输入框默认圆角 `12px`，高度不低于 `42px`。
- 主流程输入框可提升到 `52px`。
- 带图标输入框必须保证图标与输入文字的基线关系稳定。
- `focus` 必须使用品牌焦点边框，不允许无焦点态。
- 辅助说明放在输入框下方，不要塞入 placeholder 替代说明。

### 5.5 按钮

| 类型 | 规则 |
| --- | --- |
| 主按钮 | 品牌渐变底，白色文字，用于主操作 |
| 次按钮 | 白底或卡片底，1px 细边框，用于辅助操作 |
| 第三级按钮 | 中性浅底，用于低优先级操作 |
| 危险按钮 | 使用 danger 语义色，避免与品牌主色混淆 |

- 常规按钮最小高度 `42px`。
- Hero 场景按钮最小高度 `52px`，最小宽度 `140px`。
- Hover 仅允许轻微上浮和阴影增强，不允许大位移或跳动。

### 5.6 状态标签与徽标

- 状态类组件统一采用胶囊形态。
- 平台标签、任务状态、辅助 chip 在结构上保持一致。
- 颜色必须基于语义：
  - 成功：绿色
  - 运行中：黄色或品牌强调色
  - 失败：红色
  - 等待中：中性灰
- 默认胶囊采用“轻底色 + 中高对比文字”的方式表达，不默认加描边；只有确实需要区分层级时才允许细边框。
- 平台标签属于“识别徽标”，不是“状态徽标”，应弱于任务状态，不得比成功/失败/运行中更抢眼。

#### 平台标签补充规范

- 视频卡中的平台标签当前支持品牌识别变体，但仍需服从 BriefVid 设计系统。
- `YouTube` 标签：
  - 浅色文字使用 `rgb(255, 0, 51)`
  - 浅色底使用中性浅底与该品牌色的轻混合
  - 深色底使用 `rgba(255, 0, 51, 0.16)`，文字提亮到偏粉红红系，避免纯红在黑底上刺眼
- `Bilibili` 标签：
  - 当前实现使用 `rgb(250, 107, 152)` 作为识别主色
  - 浅色底使用中性浅底与该品牌色的轻混合
  - 深色底使用 `rgba(250, 107, 152, 0.16)`，文字提亮到偏浅粉
- 平台标签不使用描边，不使用高饱和纯色铺底，不允许做成“贴纸感”品牌块。
- 暗色主题下平台标签应关闭浅色模式里的高光/描边式修饰，优先保证干净和可读。

### 5.7 卡片

- 基础卡片使用浅边框、轻背景、圆角和中性阴影。
- 信息卡遵循“标题 -> 主要内容 -> 辅助说明”的三层结构。
- hover 只增强边框或轻微阴影，不使用重发光。

### 5.8 视频卡片

- 视频卡片是内容消费主单元，优先突出封面与标题。
- 标题最多两行，必须截断。
- 上层信息展示平台、状态、时长。
- 下层信息展示更新时间、处理结果、辅助描述。
- hover 时封面允许轻微缩放，但不能造成布局抖动。

### 5.9 设置页组件

- 设置导航项要能同时承载标题与一句解释性描述。
- 设置页 Hero 负责说明“当前设置模块的目的”，不要堆按钮。
- 表单分组必须有清晰标题和简短说明。
- 环境、日志、CUDA 等技术信息必须保持卡片化与分组化，避免裸列表。

### 5.10 进度、时间线与任务反馈

- 进度条默认使用品牌渐变。
- 成功和失败状态的进度条使用专属语义色。
- 时间线项必须包含标记、标题、元信息、说明内容四部分。
- 任务状态更新优先体现当前步骤，其次才是历史细节。

### 5.11 空状态、错误状态

- 空状态需要同时说明“当前为什么为空”和“用户接下来能做什么”。
- 错误状态要明确是网络、后端、更新还是用户输入问题。
- 错误块用 danger 语义底色，但正文仍要保持可读，不要整块纯红。

### 5.12 更新弹窗

- 遵循 `头部 / 状态区 / 详情区 / 底部操作区` 四段结构。
- 当前版本、新版本、发布日期、更新日志应层次清晰。
- 下载中和安装中必须有不可误解的状态文案。
- 在小屏下，底部按钮改为竖排全宽。

## 6. 交互规则

### 6.1 状态反馈

- 所有可点击元素必须有 hover、active、disabled 至少三种状态。
- 触发异步操作后必须立即给出视觉反馈，不能只靠最后结果更新。
- 加载中优先使用文本状态、进度条或 skeleton，而不是无限空白。

### 6.2 焦点与键盘

- 必须保留全局 `:focus-visible`。
- 图标按钮、导航项、筛选 pill、弹窗关闭按钮都应可键盘访问。
- 弹窗打开后，焦点应落在弹窗内的主要操作或关闭控件上。

### 6.3 滚动与层级

- 侧栏和内容区的滚动逻辑必须清晰，不允许双重滚动造成迷失。
- sticky header 只用于帮助定位，不应遮挡主要内容。
- 遮罩层、对话框、标题栏、浮层的 z-index 必须稳定，避免临时魔法数字。

## 7. 文案与图标规范

### 7.1 文案

- 操作按钮优先使用动词短语，例如“开始处理”“检查更新”“立即重启安装”。
- 状态文案优先回答“当前状态”和“下一步动作”，避免空泛的技术口吻。
- 技术信息出现在系统页或日志页时，可以保留术语；主流程页尽量面向普通用户表达。

### 7.2 图标

- 常规导航图标建议 `18px`。
- 状态图标建议 `20px` 到 `24px`。
- 图标风格保持线性、圆角端点、一致笔画粗细。
- 图标只做辅助识别，不能代替文字标签。

## 8. 可访问性要求

- 文本与背景对比度应满足基础可读性，正文与主要操作优先满足 WCAG AA。
- 所有仅图标按钮必须有 `aria-label`。
- 表单项必须有关联标签。
- 错误提示不能只依赖颜色区分，必须同时有图标或文本。
- 触控或点击热区建议不小于 `40px`，当前按钮最小高度为 `42px`。

## 9. 前端实现约束

### 9.1 Token 优先

- 新样式优先复用 `:root` 中现有变量，不要直接写死颜色值。
- 只有系统行为特殊值可以例外，例如 Windows 关闭按钮 hover 红色。
- 平台识别色属于少数可以直接写死的例外值，但只能用于局部徽标，不可扩散为页面级 token。

### 9.2 主题一致

- 新增组件必须同时验证浅色和深色主题。
- 若新增 token，需同步补齐浅色与深色定义。

### 9.3 结构一致

- 新页面优先复用现有壳层类名和布局模式，例如：
  - `.content-frame`
  - `.page-header`
  - `.grid-card`
  - `.primary-button`
  - `.secondary-button`
  - `.settings-nav-item`
  - `.update-dialog`

### 9.4 内联样式边界

- 仅允许把真正动态的值放进内联样式，例如进度宽度、计算位置。
- 颜色、字号、圆角、阴影不应通过内联样式散落在 JSX 中。

### 9.5 新组件验收要求

- 具备默认、hover、focus、disabled 四类状态。
- 具备浅色与深色两套表现。
- 具备中文文案和无障碍属性。
- 在 1400px、1024px、768px、640px 下至少做一次视觉检查。

## 10. 设计评审清单

- 首屏是否仍然突出“输入链接并开始处理”这个主动作。
- 页面标题、描述、主要操作、辅助信息是否层级清晰。
- 是否复用了现有 token、按钮、卡片、标签与表单规则。
- 深色主题下是否仍然清晰、克制、不过度发灰。
- 响应式收缩后是否仍然可操作，不出现重叠、裁切、双滚动。
- 错误、空状态、加载状态是否都能解释“现在发生了什么”。
- 仅图标按钮、状态控件、弹窗是否都具备可访问标签。
